# 프로젝트 규칙 — 단일 진실 소스 (Claude Code가 매 턴 자동으로 읽음)

> 이 파일이 이 프로젝트의 **유일한 규칙 문서**입니다. 다른 프롬프트들(steps·run·idea)은 규칙을 다시 적지 않고 "이 규칙대로"라고만 말합니다.
> 자율 진행(원샷/오토파일럿)의 첫 행동에서 이 파일을 작업 폴더에 `CLAUDE.md`로 복사하면, 이후 모든 규칙이 자동으로 적용됩니다.

## 스택 (고정)
- 이 프로젝트는 Next.js(App Router)와 React, Tailwind CSS와 shadcn/ui로 화면을 만들고, 데이터와 인증은 Supabase(@supabase/ssr)를, 배포는 Vercel을, 패키지 관리는 npm을 사용한다.
- 버전 번호, 모델 이름, CLI 문법은 기억에 의존하지 말고 오늘 날짜 기준으로 웹을 검색해서 최신 안정 버전을 확인한 뒤 사용한다. 검색으로 확정한 중요한 사실은 decisions.md 또는 docs/research/ 폴더에 기록해 나중에 다시 찾지 않게 한다.
- 이 프로젝트는 Next.js 16.2.10 LTS + React 19.2.4 + Tailwind CSS v4 + shadcn/ui(CLI v4, radix base)로 스캐폴딩되었다(2026-07-04 확인). `AGENTS.md`에 이 Next.js 버전이 학습 데이터 시점과 달라진 breaking change 안내가 있으니, API를 쓰기 전에 `node_modules/next/dist/docs/`의 관련 가이드를 먼저 확인한다.

## 환경변수 (이름 고정 — 모든 파일에서 동일하게)
- `NEXT_PUBLIC_SUPABASE_URL` 과 `NEXT_PUBLIC_SUPABASE_KEY`(publishable 키)는 클라이언트에 노출되어도 된다. 데이터 보안은 키가 아니라 RLS가 담당한다.
- `ANTHROPIC_API_KEY` 는 서버 전용이다. 클라이언트 번들과 git 저장소에 절대 들어가면 안 된다.
- 환경변수는 로컬의 `.env.local` 과 Vercel 프로젝트 설정 **양쪽 모두**에 등록해야 한다. 한쪽만 등록하면 배포된 사이트에서 동작하지 않는다. 사용자에게 Vercel 등록을 ▶ 표시로 안내한다.

## 품질 (예외 없음)
- **placeholder 금지**: 가짜 데이터, 빈 껍데기 화면, "준비 중" 표시, 기능 생략으로 작업을 때우지 않는다. 요청받은 기능은 실제로 동작하는 상태까지 만든다.
- **증거 없이 "됐다" 금지**: "동작합니다"나 "고쳤습니다"라고 말하기 전에, 빌드 통과 결과, 실행 화면, 명령 출력, grep 결과 같은 실제 증거를 먼저 보인다.
- 화면은 모바일(390px)과 데스크톱(1280px) 양쪽에서 깨지지 않아야 하며, 모든 화면에 로딩 중·데이터 없음·에러·권한 없음 상태를 갖춘다.

## 디자인
- 색, 간격, 글꼴은 디자인 토큰(핸드오프, `/design-sync`, 또는 `design/sot.html`에서 옮긴 Tailwind 설정)만 사용한다. 화면 코드에 임의의 색상값을 직접 쓰지 않는다.
- 색을 바꿀 때는 토큰 한 곳만 수정해서 전체 화면에 반영되게 한다.

## 보안
- 비밀 키는 서버(Route Handler)와 환경변수에만 둔다. 로그인이 있는 프로젝트는 모든 테이블에 RLS 정책(`auth.uid() = user_id`, 공개 데이터는 명시적 공개 정책)을 적용하고, 클라이언트에서 오는 값은 신뢰하지 않는다.
- AI 기능의 중간 다리는 **Next.js Route Handler(app/api/ai)를 기본**으로 한다. Vercel에 함께 배포되므로 따로 올릴 것이 없다. (Supabase Edge Function은 별도 요청이 있을 때의 대안.)

## 자율 진행 (원샷/오토파일럿일 때)
- 멈추는 경우는 딱 두 가지다: (a) 시작할 때 크레덴셜 봉투를 받을 때, (b) 사람만 할 수 있는 행동이 필요할 때. 그 외에는 멈추지 않는다.
- 사람의 행동이 필요하면 `▶ 지금 해주세요:` 블록으로 어느 사이트의 어느 메뉴에서 무엇을 누르고 무엇을 붙여넣어야 하는지 정확히 안내하고, 사용자가 "완료"라고 하면 즉시 이어서 진행한다.
- 지시가 모호한 부분은 멈춰서 묻는 대신 합리적인 기본값을 채택하고, 무엇을 왜 그렇게 정했으며 어떻게 되돌릴 수 있는지 decisions.md 에 한 줄로 기록한다.
- 막히면 멈추지 말고 서브에이전트에게 표준 해결 방식을 조사시켜 시도한다. 그래도 안 되면 blockers.md 에 기록하고 우회한 뒤 다음 작업으로 넘어간다. 한 지점에 30분 이상 쓰지 않는다.
- 차별점 유지: 기획에서 정한 "이 제품이 평범하지 않은 이유 한 줄"이 단계를 거치며 깎이고 있지 않은지 매 단계 점검하고, 깎였으면 같은 턴에 복구한다.

## 작업 방식 / 상태
- 코드를 바꾸기 전에 어떤 파일을 어떤 순서로 다룰지 계획을 먼저 말하고, 승인을 받으면 실행한다(자율 모드에서는 자동 승인으로 간주).
- 조사와 검수는 서브에이전트(researcher/reviewer)에게 위임해서 메인 대화를 깨끗하게 유지한다.
- 매 단계가 끝나면 project-state.yaml 에 지금까지 한 일과 다음에 할 일을 갱신한다. 세션이 끊겨도 이 파일을 읽고 이어서 진행할 수 있어야 한다.
